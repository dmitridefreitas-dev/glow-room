Follow the money — how a purchase becomes access

This is where the most services touch each other. Each box names which service does the work.

01
Web app + Stripe

Pay

Member checks out on Stripe's hosted page. Card never touches your servers.

02
Stripe → Web app

Webhook fires

Stripe pings your /api/webhooks/stripe the instant payment confirms.

03
Supabase

Provision

App writes the enrollment + a single-use access code to the database.

04
Resend

Email the code

The code is emailed to the member (and also shown on /welcome).

05
Bot → Supabase → Discord

Verify & unlock

/verify burns the code and the bot grants the Discord member role.

Two more flows run quietly alongside: every page view streams to PostHog for analytics, and (once Phase 6 lands) crashes stream to Sentry and reminders go out through OneSignal. A refund or cancel runs step 3–5 in reverse — Stripe's webhook tells the app to revoke the enrollment and the bot removes the Discord role.

Each service in plain English

Supabase

Core · database, auth & storage Core · wired

What it does

It's three things in one: a Postgres database (members, cohorts, check-ins, badges, access codes), a login system (email + password), and private file storage for check-in photos. Row-Level Security means people can only read their own rows.

Why it's here

It's the source of truth for who has paid and what they've done. Everything else reads from or writes to it.

The three keys

  • ANON_KEY — safe for the browser; only sees what RLS allows.
  • SERVICE_ROLE_KEY — server-only superkey; bypasses RLS to provision members. Never ships to the browser.
  • URL — the address of your database.

Stripe

Core · payments Core · wired (test mode)

What it does

Collects payment — the $18 one-time cohort and the $9/mo membership — on Stripe's own secure checkout page. A webhook tells your app the moment a payment succeeds, which is what triggers provisioning. The billing portal lets members cancel themselves.

Why it's here

Handling cards yourself is a compliance nightmare. Stripe owns that risk; you just react to "payment succeeded."

The keys

  • SECRET_KEY — server-side, creates checkouts.
  • WEBHOOK_SECRET — proves a webhook really came from Stripe, not a faker.
  • MEMBERSHIP_PRICE_ID — which price the $9/mo button charges.
Before launch: switch from test keys to live keys and create the production webhook (Phase 6).

Resend

Communication · email ● Live & wired · 1 action

What it does

Sends transactional email. Right now that's the "You're in" email with the Discord access code, sent automatically after someone pays.

Why it's here

Members need their access code even if they close the tab. Email is the durable copy; the /welcome page is the instant copy.

Status from the live check

Key valid and the send code is wired. But no verified domain yet — so emails currently only reach your own Resend account address.

Before launch: verify your sending domain in Resend and change the from: address in lib/email.ts.

Discord

Communication · community access ● Live & wired

What it does

Runs the gated community. The bot's /verify command checks a member's code and gives them the "Glow Member" role; if they later refund or cancel, your app calls Discord to remove the role automatically.

Why it's here

The community ("doing it together") is the product's hook. Discord is where it lives; the role is the on/off switch for access.

The keys

  • BOT_TOKEN — lets your bot act on the server.
  • SHARED_SECRET — a password between the bot and your API so only your bot can redeem codes.
  • GUILD_ID / MEMBER_ROLE_ID — which server and which role.

Check confirmed: bot GlowRoomBot, server TheGlowRoom, role Glow Member all found.

PostHog

Growth · product analytics ● Live & wired

What it does

Watches how people actually use the app — page views, clicks, and funnels (how many who hit "join" actually pay; where people drop off across the 30 days). It ties activity to a logged-in person so you can measure retention.

Why it's here

Knowing where members quit is what lets you fix the product — it's the behavioural data the business plan calls the "moat."

Good to know

  • The key is a public project key — it's meant to ship to the browser.
  • It only captures usage events, never card or password data.

Check confirmed: a test integration_check event was accepted.

OneSignal

Growth · push notifications ● Live · not wired yet (Phase 6)

What it does

Sends push notifications — the planned "Day 8 is the hard day" nudge and daily reminders to come back and check in, even when the app is closed.

Why it's here

Reminders are how a daily-habit product keeps people from drifting. This is a retention lever, deferred until after launch.

Status from the live check

Account "TheGlowRoom App" is live with 0 subscribed devices — expected, because nothing collects subscribers yet.

Phase 6: add a service worker to collect subscribers + a scheduled sender to fire the nudges.

Sentry

Reliability · error monitoring ● Live · not wired yet (Phase 6)

What it does

Catches crashes and errors in production and alerts you with the exact stack trace and the steps that caused it — so you hear about bugs from Sentry, not from an angry customer.

Why it's here

Once real money and real members are involved, silent failures cost trust. Sentry is your early-warning system.

Status from the live check

The DSN works — a labelled test event was accepted. But no app code reports to it yet (just a placeholder in app/error.tsx).

Phase 6: install @sentry/nextjs so server & browser errors auto-report.
